Introduction

The application we have developed is called maxwallet. As explained in the abstract, the aim of maxwallet is to allow users to create a "web wallet", an online interface that allows them to access their MaxCoin and to send it to other users or merchants. The advantage of using a web wallet is threefold: users don't have to download a piece of software which, additionally, downloads the blockchain - the public ledger of all the transactions within MaxCoin; users can access their wallet from anywhere, worldwide, with an Internet connection; and, finally, it allows additional functionality to be included, such as the Standing Orders present within maxwallet, a feature that the desktop client doesn't have.

Currently only a small handful of cryptocurrencies have web wallet services and it will be a great boon to MaxCoin once the service is launched. Additionally, we have taken great care to ensure the codebase is written in a modular manner so that it can be ported with ease to other currencies. Due to the financial nature of the web application, we have been vigilant to ensure that our code is as secure and well written as possible. If the service was to be hacked it would immediately lose it's credibility and become a detriment to the currency.

Existing services for storing and handling cryptocurrency online include blockchain.info and dogechain.info; these services typically allow a user to send and receive currency. With maxwallet, we will build upon this functionality by including the option to create regular payments to an address and to submit payment requests, much like a regular bank. We will also fully integrate with the upcoming MaxCoin Android wallet by allowing a user to interface with the web application using QR codes that we will both generate and read, using the client's camera.

Angular JS

For this project we decided that we would use Angular JS. Angular is an open-source javascript MVC framework maintained by Google that assists in the making of single page applications. From doing research and experimenting with Angular, as well as other frameworks such as Ember JS, we concluded that Angular was the best fit in aiding us make a single page dynamic web application. Other frameworks deal with HTML's shortcomings by abastracting away HTML, CSS and/or javascript or by providing an imperitive way for manipulating the DOM. However none of these methods address the problem that HTML was simply not designed for dynamic views. Angular JS overcomes this by allowing you to extend the HTML vocabulary of your application.

The first thing we looked to do with Angular was to enable navigation between pages without any reloads i.e. lay the foundations for a single page app.

In Angular JS directives are markers on a DOM element (such as an attribute, element name, or CSS class) that tell Angular JS' HTML compiler to attach a specified behavior to that DOM element or transform the DOM element and its children. One of the directives that comes bundled with Angular is ng-view. When you set up the routing in angular with ng-view, you specify the HTML template Angular should load when asked for a particular URL. You typically add the ng-view attribute to an empty div which is to hold the main body of your content, and this is where Angular will inject the HTML template specified by your routing.

We however found the ng-view directive to be a bit limiting as it is not possible to nest views within each other with it. However, we went away and researched possible solutions and ended up quickly coming accross a popular Angular module called ui-router. ui-router gave us exactly what we were looking for; the ability to nest views with each other. ui-router works very similarly to Angular JS' router but it focuses purely on the state of the application. The routing is logically pretty similar to Angulars default router except now you're providing templates for particular states, and within the application you link to states rather than urls. However the thing we're intersted in is the abilty to construct a state hierachy, i.e. parent/child states aka nested states. In our application for example, this enables us to have a wallet state, and then have child states of wallet such as send, recieve etc.

Once we had got our heads around the basic components of Angular JS and the ui-router specific directives for routing, we were left with everything we needed to create a basic 'one page application' with nested states. One thing we struggled with was making it so when you were in some child state of a state such as wallet (a state for when the wallet tab in the navbar was clicked), the wallet tab was still toggled as selected in the nav bar. We looked at how the ui-router sample application did this, however we found out that it too was using the same method as we but with no success. After much searching, we finally found the fix and also ended up submitting the fix as a pull-request the ui-router git repository.

The next thing we set out to do was create a reusable Angular JS modal window directive. One of the idiums of Angular JS is that directives should be generic and reusable. A modal window in most cases can be said to have 3 components; the overlay behind the window, the window itself and the content. So ideally the directive that we construct for the modal window will always contain the same overlay and window but it'll just be the content inside the window that we change. Angular directives have a mind-boggling amount of options; one of them transclude. When transclude is set to true, and a template is provided within the directive, the content inside the element with the directive attribute, in this case our modal window directive attribute, is inserted into the element in the template with the ng-transclude attribute (only one of these is allowed in the template). This is exactly what we want. In our template we can put the overlay class and a window class, and specify with the ng-transclude attribute where we want our custom content to be inserted inside our template. To show and hide the modal window we can use the native angular directive ng-show which when assigned an expression which evaluates to true will hide itself and its children. If we have a variable such as modalShown which is initially set to false within our current scope, we can set a button to perform a function call which toggles the value of modalShown using the attribute ng-click. N.B that the difference between on-click and ng-click is that ng-click performs actions on the current scope.

Controllers in Angular JS give us the really nice feature of encapsulation; anything in the HTML contained within an element with the attribute ng-controller="someController" will have access to its scope. Controllers are where you set up the intial state of the $scope object. In the case of the modal window, it's where we we set intially set modalShown equal to false. The element with the ng-show attribute will need to evaluate the modalShown value on the "someController" scope for everything to work correctly. This fundamentally means that it needs to be contained within an element with the attribute ng-controller="someController".

Now that we had created a modal window directive that we can populate with arbitrary content, it seemed logical for us to create a login form. The login form was to have an input field for both an email address and passsword. We decided to override the default HTML5 form validation error messages with our own as the HTML5 error messages didn't fit in that well theme wise with the rest of our site. Whenever you create a form within an angular application, it is actually an instance of an angular Controller FormController. Forms of instance FormController have their own scope which has a lot of useful properties on it. FormController lets you use attributes to specify whether a particular field is required, whether it has been left untouched, maximum and minimum valid length as well as a few other useful attributes that will set corresponding values on the scope indicating validity. This means that we can easily create nicely styled error messages by using ng-show to show and hide error messages based on what the user has entered. For example we could use ng-show in the following way to display to display unhide an element if the user has entered data in an input field and the data isn't valid where userForm is the name of the form and password is the name of the text field: data-ng-show="userForm.password.$invalid && !userForm.password.$pristine". We also can check whether the entire form is valid using userForm.$invalid. Using this property we were able to disable the login button button of the login form until the user has entered a valid email and password. We have used similar form validation elsewhere in our application for other forms. Interestingly on the most up to date build version of angular, native number validation is broken, so we're likely going to have to implement the validation ourselves or even try and fix the native angular validation and submit a pull-request.

When wanting to recieve some bitcoin/maxcoin you give someone an address to send the bitcoins to. It is useful sometimes to show someone a qrcode for them to scan. Additional paramters can be address to this address such as a label and amount. We thought we could demonstrate Angular's power by creating a qrcode that updated instantly when the user enters additional information such as a label or an amount. Angular works really well with existing javascript libraries. We created a directive to nicely wrap up the use of the library called qrcodejs. The nice thing bout using a directive for logic such as this, is that we don't have to query the DOM for the particular element we want to give to the qrcode library; the element is passed to us as a paramter inside the directive. Making the QRCode update when the user input changed prooved to be relatively straight-forward. We can bind the value contained in each input field to two values on the scope using the ng-model as an attribute on each input field element. We can use an angular function called $watchCollection to watch the values on the scope bound to the input fields. This function will fire everytime either of these values change. We can then update the QRCode element with the new information from within this function.

Typically MVC apps make calls to the API for data which the app is going to display. For the most part in our application, we are not making that call to our API as it currently does not exist yet. But instead, we add some dummy data to the scope and use angular to dynamically template the HTML for us. In the case of our address book, our data will consist of a name and a coressponding payment address. We can use the angular ng-repeat in the following to template some html for us: data-ng-repeat="data in tabledata". Whatever is inside the element that has this attribute will be repeated for every item in tabledata. Inside this element we can then use the curly bracket angular notation {{data.name}} to get the name of each item and put it in our table. For the server side of this project, the table data on the scope will be data returned by a call to our API.

As templated HTML is just a view of the data, in Angular when you change the data, the view of the data changes also. This means, that if we want to add data to the table, we just need to add items to the data model. This in Angular is fairly trivial to implement. We again bind the user input fields to the scope using ng-model. We then also create a function on the scope that when called pushes the values in the input fields into the json object holding data and then clear the variables bound to the text fields. When the server side is implemented, we will also have to send this information to the server for it to put in our database.

In the overview tab of our wallet, we wanted it so that the current value in dollars of your coins is displayed. We created a Angular service that makes a cross site ajax call to a service hosting a json object with up to date price information. This service in true angular fashion can be injected as a dependency in any controller and its data accessed. Creating global services such as these is encoruaged over using global variables. We then parse the data and update a varible bound to a label in our wallet.overview so that the total value in dolars of the wallet's contents is displayed. Currently the total amount of coins is just a value on the scope so that it's really simple for us to update our application when we have an API.

We decided to wait until the server was ready before we implemented a 'logged in'and 'logged out' state as there is a fair amount of functionality that needs to be added server side to properly implement such a thing. So for the time being we just wanted to display a nice looking toast when the login form say was submitted. For the toasts we used an angular module called 'toaster' which nicely wraps jquery toasts to give you a directived that can be reused over and over. We had to a make a few small modifications to this module to make the output of the XHTML compatable. However when we had sorted this and the dependency injection out we could add a function on the scope that could be called by any button to display a toast. For example, when the user pressed login on the login form, the modal window closes and a 'successful login' toast is displayed

Responsive Design

The website we have created is fully responsive, working from all sizes of device from a widescreen, full HD screen (and larger) to a small mobile phone. The style sheets employed describe the widths as either percentages, for wrapping elements and containers, or ems, for anything related to text. Due to the use of a "sticky footer" and a "floating header", we have used pixel values in these areas; we have, however, used the @media CSS attributes to allow these to scale with the screen size.

Certain components display completely differently when presented at different screen resolutions. One example of this is the navigation bar in the page header: this is normally a horizontal bar containing navigation items but, at screen resolutions below 768px, this is displayed as a single menu icon. When the menu icon is clicked it toggles a drop down menu containing the navigation links, conserving space on mobile devices. Another example is the tables within the Wallet section of the website: this section features a number of wallets containing information on transactions and addresses, each of which are stored in a tabular format. The tables are styled such that when they begin to overflow the surrounding element they contract and display a horizontal scroll bar instead.

We support both of these browsers, including very old versions of them (we've tested back to version 10 for both, see below) along with Opera and a host of Linux-specific browsers, and are compatible with all of them. Search Engine Optimisation

Our website is being designed for real-world deployment and, as such, we have included features such as Search Engine Optimisation. The index page of the website contains meta tags with keywords, a description and the authors. The index page injects the content of other pages into itself, meaning it will be the most frequently crawled and thus the most important. When the back-end is implemented, meta tags for other pages can be modified using JavaScript. We have also submitted a robots.txt file; this is used by web-crawlers such as Googlebot to determine where to crawl and index pages, along with where not to go. The robots file also contains a link to our XML Sitemap. The Sitemap, sitemap.xml, describes the change-frequency and the index priority of the pages within the website, along with allowing isolated pages to be indexed. Both our robots file and the Sitemap conform to the relevant standards and could be submitted to most modern search engines as they are.

Once the back-end of the website has been implemented, we will also include automatically-updating HTML "snapshots" of the dynamic content on the website, or that injected using JavaScript. This will allow these pages to be crawled by a traditional web-crawler and they will be added to the Sitemap when they are in place.

SVG

We employed Inkscape to create 2 SVG graphics. One of these, a camera icon, was created for use as a button on the Send Funds section of the Wallet; the other, an outline of an envelope, was imported into GIMP and used an image on the Contact page.

The camera icon was created solely within Inkscape. It was created by first making the lens using concentric circles, each with a small gap between it and the outer circle, varying the width of their strokes to create the effect of a zoomed camera lens. The centre of the lens was then given a radial blur and a highlight, using a low-opacity white layer that has been shaped using a Boolean difference operation. The rest of the camera was made using rectangles with a curved stroke, in varying shades. A line across the camera body was added for some interest and given a shadow, by duplicating it, moving it slightly down and right, then lowering the opacity. The same was done for the camera lens to create some depth. Finally, a highlight was given to the camera viewport in the same way that the highlight to the lens was created.

The envelope was a much simpler creation as it was to be modified within GIMP. The envelope was created by using the rectangle tool to create the basic shape, again with a rounded stroke to give it a softer appearence. The shadows and the top flap were then created by using the rulers to mark the edges of the envelope and using the straight line tool.

GIMP

To demonstrate our ability to use GIMP, we created two images for the Contact section of the website. The first of these images, as described above, is a coloured envelope, to be displayed next to the business address as a postal icon. This was created by importing the SVG outline, colouring it and then cutting out the letter so that it sits within the envelope. The letter was cut out by using the Magic Wand tool to select the shadowed inside of the envelope, then, taking the inverse, and hiding the part of the letter within this selection. A highlight was then added by using a large, feathered brush set to a low opacity. The second of these images features Luke and Josh, framed within a circle each. This was created to add a personal element to the website. The image utilised a circular selection to cut out the faces from two photographs and placed on a transparent background. It was kept deliberately simple to be continuous with the professional, clean theme of the website.
